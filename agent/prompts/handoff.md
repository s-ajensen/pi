Before we hand this to the implementer: ${@:-the work we just ratified}.

The implementer has this whole arc in context — what it lacks isn't
information, it's settledness: experience says the first response back will
re-open decided questions as fresh forks, usually resting on a premise that
dissolves under inspection. Think one step ahead. Anticipate the
interrogations this work will produce and answer them now:

1. **Probe before asserting** — where a likely question hinges on a checkable
   fact (source, docs, tests), check it now so the answer is a fact, not a
   position. If probing turns up a *real* trap, flag it as a legitimate stop —
   distinguish these sharply from the dissolved forks.
2. **Name what's settled** — the decisions this work executes, stated as
   closed: ratified, checkpoint spent, not up for re-litigation.
3. **Tripwires** — anything corrected this session that this work gives an
   opportunity to regress on.
4. **Gates** — the explicit points where execution stops for eyes. State
   plainly: everything between gates is standing authorization; stop only at
   the listed gates or on a genuine surprise.
